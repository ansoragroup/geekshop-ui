<SectionTitle>Touch Targets</SectionTitle>

WCAG 2.5.5 requires interactive elements to be at least 44x44px. The library enforces this across all interactive components:

```scss
// Internal pattern used in Button, TabBar, etc.
.button {
  min-height: 44px;
  min-width: 44px;
  // Padding expands the touch target even when visual content is smaller
}
```

**The padding trick** -- For small visual elements (icon buttons, checkboxes), the touch target is larger than the visible element:

```scss
// A 24px icon with a 44px touch target
.iconButton {
  width: 24px;
  height: 24px;
  padding: 10px; // (44 - 24) / 2 = 10px on each side
  margin: -10px; // negative margin so it doesn't affect layout
  cursor: pointer;
}
```

Components that implement this pattern:
- **QuantityStepper** -- Plus/minus buttons
- **Checkbox** -- Checkbox input
- **Radio** -- Radio input
- **Switch** -- Toggle switch
- **Rating** -- Star tap targets
- **TabBar** -- Tab items
- **ActionBar** -- Action buttons

<SectionTitle>Focus Management</SectionTitle>

### useFocusTrap Hook

Overlay components use the `useFocusTrap` hook to trap keyboard focus inside the overlay when it is open:

```tsx
import { useFocusTrap } from '@ansoragroup/ui'

function MyOverlay({ isOpen, onClose }) {
  const containerRef = useFocusTrap(isOpen, {
    onEscape: onClose,    // close on Escape key
    restoreFocus: true,   // return focus to trigger on close
  })

  if (!isOpen) return null

  return (
    <div ref={containerRef} role="dialog" aria-modal="true" tabIndex={-1}>
      <button onClick={onClose}>Close</button>
      <div>Overlay content...</div>
    </div>
  )
}
```

**How it works:**
1. When `isOpen` becomes `true`, focus moves to the first focusable element inside the container
2. `Tab` and `Shift+Tab` cycle within the container only
3. `Escape` calls the `onEscape` callback
4. When `isOpen` becomes `false`, focus returns to the previously focused element (if `restoreFocus` is true)

**Components using useFocusTrap:**
- Popup
- Dialog
- BottomSheet
- QuickBuyPopup

<SectionTitle>Screen Reader Support</SectionTitle>

### Live Regions

The `Toast` component uses `role="alert"` and `aria-live="assertive"` so screen readers announce notifications immediately:

```tsx
// Internally, Toast renders:
<div role="alert" aria-live="assertive" aria-atomic="true">
  Item added to cart!
</div>
```

### Image Alt Text

Components that display images follow consistent alt text patterns:

```tsx
// ProductCard — uses product title
<img alt="RTX 4070 Graphics Card" />

// ProductImageGallery — uses index
<img alt="Product image 1 of 5" />

// Avatar — uses user name
<img alt="John Doe" />
```

### Hidden Decorative Elements

Decorative elements (icons next to labels, background patterns) use `aria-hidden="true"`:

```tsx
<span aria-hidden="true">★</span>
<span>4.5 out of 5 stars</span>
```

<SectionTitle>Reduced Motion</SectionTitle>

All animations respect the `prefers-reduced-motion: reduce` media query. When this preference is set, transitions and animations are disabled:

```scss
// Pattern used across all components
.element {
  transition: transform 0.25s ease;
}

@media (prefers-reduced-motion: reduce) {
  .element {
    transition: none;
  }
}
```

Components affected:
- **BottomSheet** -- Slide-up animation disabled; appears/disappears instantly
- **Toast** -- Slide-in animation disabled; appears/disappears instantly
- **Popup/Dialog** -- Scale/fade animation disabled
- **Button** -- Press scale feedback disabled
- **Loading** -- Spinner animation simplified (no rotation)
- **Skeleton** -- Shimmer animation disabled
- **Swipe** -- Swipe transitions disabled
- **CountdownTimer** -- Digit flip animation disabled

<SectionTitle>Testing Accessibility</SectionTitle>

### Storybook a11y Addon

The library includes `@storybook/addon-a11y` which runs automated accessibility checks on every story. Open any story in Storybook and check the **Accessibility** panel tab to see violations and warnings.

The addon checks for:
- Color contrast violations
- Missing ARIA labels
- Invalid ARIA attributes
- Focus order issues
- Missing alt text

### Manual Testing Checklist

When creating or modifying components, verify:

1. **Keyboard only** -- Navigate the component using only Tab, Shift+Tab, Enter, Space, Escape, Arrow keys. Every action should be possible without a mouse.

2. **Screen reader** -- Test with VoiceOver (macOS) or NVDA (Windows):
   - Are interactive elements announced with their role? ("button", "link", "checkbox")
   - Are labels read correctly?
   - Are state changes announced? ("checked", "expanded")

3. **Zoom** -- Test at 200% browser zoom. Content should reflow without horizontal scrolling on mobile viewports.

4. **High contrast** -- Test with Windows High Contrast Mode or `forced-colors: active` media query.

5. **Reduced motion** -- Enable reduced motion in OS settings and verify no animations play.

### Automated Tests

Component test files include accessibility assertions:

```tsx
// Example from Button.test.tsx
it('has correct ARIA attributes when disabled', () => {
  render(<Button disabled>Click</Button>)
  const button = screen.getByRole('button')
  expect(button).toHaveAttribute('aria-disabled', 'true')
  expect(button).toBeDisabled()
})

// Example from Popup.test.tsx
it('traps focus inside the popup', () => {
  render(<Popup open onClose={() => {}}>Content</Popup>)
  const dialog = screen.getByRole('dialog')
  expect(dialog).toHaveAttribute('aria-modal', 'true')
})
```

<SectionTitle>Accessibility Props Reference</SectionTitle>

All components accept standard HTML accessibility attributes via prop spreading (`...rest`). Common overrides:

```tsx
// Override the auto-generated aria-label
<Button aria-label="Add RTX 4070 to cart">
  Add to Cart
</Button>

// Link a form input to an external label
<Input id="email" aria-describedby="email-help" />
<span id="email-help">We will never share your email.</span>

// Mark a decorative component
<Avatar aria-hidden="true" src="/decorative.jpg" />
```
